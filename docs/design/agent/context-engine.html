EchoMem Agent / Context Engine Design

EchoMem Agent Context Engine

本文定义 echomem-agent 的上下文工程方案：把稳定规则、任务环境、记忆检索、会话历史、工具观察和当前请求组织成可缓存、可压缩、可解释、可演进的 Agent 上下文。

目标：通用任务型 Agent 文档：docs/design/agent/context-engine.html 重点：顺序、预算、缓存、记忆 参考：OpenClaw / Hermes / Prompt Caching
目标 阶段 原则 组成 顺序 分层 生命周期 预算 检索 工具 安全 落地 参考

1. 设计目标

Context Engine 是 Agent 的“工作记忆编排器”。它不只是把字符串拼起来，而是决定模型每一轮能看见什么、先看见什么、哪些内容能复用缓存、哪些内容必须被检索、哪些内容只能作为调试信息保留。

通用任务能力

覆盖多类任务

支持问答、代码、文档、计划、工具调用、多步执行、长期项目协作。上下文必须能表达目标、约束、证据、状态和下一步动作。

长期记忆接入

EchoMemory 原生

把 EchoMemory 当作事实和记忆来源，通过 session、retrieval、refs、commit 形成闭环，不在 Agent 侧复制一套隐式记忆系统。

缓存友好

稳定前缀优先

借鉴 Prompt Caching 思路，把不变内容放在前面，把时间、检索结果、工具结果和当前用户输入放在后面，减少重复推理成本。

核心判断：Agent 能不能“完成各类任务”，很大程度取决于上下文能否稳定表达任务环境，并在动态信息进入时不冲散长期规则和关键目标。

2. 阶段划分

Context Engine 不一次性做成全能力系统。第一阶段只服务当前“纯对话 Agent”：用户发消息，Agent 结合系统规则、会话历史和 EchoMemory 检索结果回答。工具、工作区状态、复杂任务计划和多 Agent 协作都放到后续阶段。

阶段目标启用上下文暂不包含完成标准
Phase 1：纯对话 Agent 让 echomem-agent 成为稳定的记忆增强聊天 Agent。 Agent Charter、Behavior Policy、Memory Contract、Session State、Retrieved Memory、Conversation Tail、Current Request。 工具调用、工作区扫描、任务计划、自动摘要、refs 写入策略、复杂 commit 策略。 每轮能写入用户消息、检索 EchoMemory、按稳定顺序组装 messages、调用模型、写入 assistant 回复。
Phase 2：长对话与摘要 支持较长会话持续进行，不因历史过长而丢失关键决策。 新增 Session Summary、Task State、token budget、context debug trace。 外部工具执行和工作区状态仍不进入模型。 超过历史预算时生成摘要，保留目标、偏好、关键事实、未完成事项。
Phase 3：工具与任务执行 从聊天扩展为能完成任务的 Agent。 新增 Tool Contract、Tool Observations、Task Plan、重要 refs 写入。 多 Agent 协作和复杂权限模型。 工具结果结构化进入上下文，重要证据写入 EchoMemory refs，失败可解释。
Phase 4：通用任务工作台 面向代码、文档、研究、自动化等多类任务形成完整上下文系统。 新增 Project Profile、Workspace State、跨任务记忆、阶段性 commit 策略。 不追求把所有历史全文塞进模型。 Agent 能在长任务中恢复状态、引用证据、管理工具结果，并保持稳定前缀缓存友好。
当前实现应优先落 Phase 1。后续章节描述的是完整 Context Engine，但每个组件都要标注它属于哪个阶段，避免第一版为了未来能力过度设计。

3. 设计原则

原则要求原因
稳定内容前置 角色、行为准则、工具契约、输出规范、记忆使用规则放在上下文最前面。 这些内容跨轮次变化少，最适合作为可缓存前缀。
动态内容后置 时间戳、当前请求、检索结果、工作区快照、工具结果放在后段。 这些内容高频变化，提前出现会破坏缓存命中，也会让稳定规则被稀释。
证据优先于回忆 检索结果必须保留来源、时间、置信度和用途；模型不能把低置信记忆当事实。 任务型 Agent 的可靠性来自可追溯证据，而不只是流畅回答。
历史可压缩 会话历史分为近期原文、阶段摘要、已完成决策、未完成待办。 长任务不能无限保留全文，需要保留“可继续工作”的最小状态。
工具结果结构化 工具观察进入专门区域，不混进自然语言历史；重要结果写入 EchoMemory refs。 工具结果是状态变更和证据，不应被普通聊天文本吞没。
上下文可解释 每轮输出可开启 debug 视图，展示哪些 memory、history、tools 被放入模型。 上下文错误通常比模型错误更隐蔽，需要可观察。

4. 上下文由哪些部分组成

echomem-agent 的上下文被拆成 12 个组件，但不会在第一阶段全部启用。Phase 1 只启用纯对话必需层；其余组件作为后续阶段的扩展点保留。

#组件内容阶段稳定性推荐 role
1 Agent Charter 身份、目标、边界、语言偏好、不可做事项。 Phase 1 几乎不变 system
2 Behavior Policy 事实性要求、安全要求、遇到不确定性时的处理方式。 Phase 1 几乎不变 system
3 Tool Contract 可用工具、调用约束、工具结果解释规则、失败重试规则。 Phase 3 低频变化 developer 或工具 schema
4 Memory Contract 如何使用 EchoMemory：检索结果只是候选证据，冲突时如何处理，何时写 refs 和 commit。 Phase 1 低频变化 developer
5 Project Profile 仓库、产品、用户偏好、团队约定、当前 Agent 配置。 Phase 4 中频变化 system 或 developer
6 Session State user_id、agent_id、session_id、当前任务阶段、未完成目标。 Phase 1 每会话变化 system
7 Task Plan 当前任务的步骤、状态、阻塞点、验收标准。 Phase 2 / Phase 3 每轮可能变化 system 或 assistant 摘要
8 Retrieved Memory EchoMemory 检索到的偏好、事实、历史决策、外部引用。 Phase 1 每轮变化 system
9 Workspace State 文件树、打开文件、git 状态、测试结果、运行环境。 Phase 4 每轮变化 system
10 Tool Observations 工具调用结果、错误、日志片段、网页或文件摘要。 Phase 3 高频变化 tool 或 system 摘要
11 Conversation Tail 最近 N 轮 user/assistant 原文，保留语气和上下文连贯性。 Phase 1 高频变化 user / assistant
12 Current Request 用户本轮原始输入和附件入口。 Phase 1 每轮必变 user

5. 推荐上下文顺序

顺序不是排版问题，而是模型注意力、缓存命中率和任务稳定性的共同约束。推荐顺序遵循“越稳定越靠前、越具体越靠后、当前用户输入永远最后”。

Phase 1 顺序：纯对话 Agent

顺序上下文层说明
1Agent Charter固定身份、目标、边界和输出风格。
2Behavior Policy事实性、安全、未知处理、引用规则。
3Memory Contract说明 EchoMemory 检索结果如何使用、如何处理冲突。
4Session Stateuser_id、agent_id、session_id、时间戳等动态会话信息。
5Retrieved Memory本轮检索到的偏好、事实和历史上下文。
6Conversation Tail最近 N 轮 user/assistant 原文。
7Current Request当前用户原始输入，永远放最后。

完整阶段顺序：Phase 2+

缓存前缀跨轮复用
1. Agent Charter 身份、目标、边界、输出风格。不要放时间、用户问题、检索结果。
system强约束
缓存前缀跨轮复用
2. Behavior Policy 事实性、安全、引用、未知处理、任务完成标准。
system强约束
缓存前缀低频变化
3. Tool Contract + Memory Contract 工具和 EchoMemory 的使用契约。工具 schema 若由 API 单独承载，则 prompt 中只保留解释规则。
developer工程约束

5. Session State + Task Plan 当前会话、任务目标、阶段、未完成事项、验收标准。

6. Retrieved Memory 按 relevance、recency、kind 分组的 EchoMemory 检索结果，带来源和置信度。

动态证据每轮变化

6. 分层格式

每一层都用明确标签包裹，减少模型误读。标签不是为了炫技，而是为了让裁剪、调试、摘要和评估都能定位上下文来源。 

[
  {
    "role": "system",
    "content": "<agent_charter version=\"2026-05-23\">...</agent_charter>"
  },
  {
    "role": "system",
    "content": "<behavior_policy version=\"2026-05-23\">...</behavior_policy>"
  },
  {
    "role": "developer",
    "content": "<tool_contract>...</tool_contract>\n<memory_contract>...</memory_contract>"
  },
  {
    "role": "system",
    "content": "<project_profile freshness=\"daily\">...</project_profile>"
  },
  {
    "role": "system",
    "content": "<session_state>...</session_state>\n<task_plan>...</task_plan>"
  },
  {
    "role": "system",
    "content": "<retrieved_memory query_id=\"...\">...</retrieved_memory>"
  },
  {
    "role": "system",
    "content": "<workspace_state>...</workspace_state>\n<tool_observations>...</tool_observations>"
  },
  {"role": "user", "content": "上一轮用户消息"},
  {"role": "assistant", "content": "上一轮助手回复"},
  {"role": "user", "content": "当前用户请求"}
]

标签规范

标签是否可缓存裁剪策略
<agent_charter>是不可裁剪，只能版本升级。
<behavior_policy>是不可裁剪，只能版本升级。
<tool_contract>是按可用工具集合生成，工具很多时只保留当前任务相关工具。
<memory_contract>是不可裁剪，保持短小。
<project_profile>部分每日或仓库变更后刷新，过长时转为摘要。
<session_state>否保留关键标识和阶段，不放冗长日志。
<task_plan>否保留未完成项、当前步骤和验收标准。
<retrieved_memory>否按分数和用途裁剪，保留来源。
<workspace_state>否只保留任务相关文件、变更、测试结果。
<tool_observations>否长输出摘要化，必要片段带行号或 URL。

7. 上下文生命周期

每一轮对话都应该经历同一条流水线，避免不同入口拼出不同上下文。 

User request
  -> classify task intent and risk
  -> open or resume EchoMemory session
  -> write current user message
  -> load stable prefix by version
  -> build project profile
  -> retrieve EchoMemory candidates
  -> inspect workspace and recent tool state
  -> summarize long history if needed
  -> allocate token budget by priority
  -> assemble ordered messages
  -> call model
  -> write assistant message and important refs
  -> optionally commit session
  -> expose context debug trace
输入侧

先写入，再检索

用户本轮消息应先进入 EchoMemory session，再用当前问题检索。这样检索层能理解最新问题与已有会话的关系。

输出侧

先验证，再写入

模型失败或工具失败时不要写入伪 assistant 成果。成功回答、工具证据、关键决策才写入 session 或 refs。

8. Token 预算与裁剪

预算策略使用“硬保护 + 分层配额 + 降级摘要”。稳定规则和当前用户请求是硬保护，其他区域按价值裁剪。

区域默认占比保护级别超预算处理
Agent Charter + Behavior Policy8%最高不可裁剪。
Tool Contract + Memory Contract10%高只保留当前可用工具和关键规则。
Project Profile8%中压缩为项目摘要。
Session State + Task Plan8%高保留目标、当前步骤、阻塞点。
Retrieved Memory24%中高按相关性、时效、来源质量裁剪。
Workspace + Tool Observations20%中高长日志摘要化，只保留错误和关键证据。
Conversation Tail14%中保留最近 N 轮，早期历史合并为摘要。
Current Request8%最高不可裁剪，附件另行摘要。
如果模型上下文窗口足够大，也不要无限塞入历史。Agent 需要的是“对当前任务有用的上下文”，不是“所有曾经出现过的文本”。

9. EchoMemory 检索设计

EchoMemory 返回的内容进入 <retrieved_memory>，并按用途分组。每条 memory 都必须说明它为什么被选入上下文。

分组用途示例
Preference 用户长期偏好、输出风格、协作习惯。 用户偏好中文、简洁、直接给可执行方案。
Project Fact 项目结构、历史设计决策、接口约束。 echomem-agent 不 import echomem.*，只通过 HTTP 访问 EchoMemory。
Task History 同一任务或相似任务的阶段、结论和未完成事项。 上一轮已经完成 model_client，下一步是 context_builder。
Evidence 文档、网页、测试、工具输出等可引用证据。 参考文章强调 Prompt Caching 需要把稳定 prompt 放在前缀。
Risk 失败记录、冲突事实、用户明确禁止事项。 不要覆盖用户未提交的本地改动。 
<retrieved_memory query_id="r-20260523-001">
  <item kind="project_fact" score="0.91" source="session:chat-001" recency="2026-05-21">
    echomem-agent 是独立工程，只通过 EchoMemory HTTP API 通信。
  </item>
  <item kind="preference" score="0.84" source="profile:user" recency="long_term">
    用户希望设计文档先讲不变部分，再讲动态上下文。
  </item>
</retrieved_memory>

10. 工具与多步任务上下文

面向通用任务的 Agent 必须能把工具调用纳入上下文循环。工具不是聊天内容，而是外部世界观察和状态变更。

对象进入上下文的方式写入 EchoMemory 的条件
工具 schema 通过模型 API tools 字段承载；prompt 中只解释使用策略。 不写入，属于静态能力。
工具调用意图 作为 assistant tool call 或 task plan 的下一步。 重要决策或用户可见动作写入。
工具结果 短结果原文进入 <tool_observations>，长结果摘要化。 影响任务结论、包含证据或状态变更时写入 refs。
错误与重试 保留错误摘要、失败命令、已尝试路径。 阻塞性错误或用户需要知情时写入。
阶段摘要 多步任务每完成一个阶段生成 <task_plan> 摘要。 阶段完成、用户确认、会话 commit 时写入。
参考 Hermes 一类 Agent 的做法，工具输出要和推理上下文分层：模型需要看见“结果是什么、是否可靠、下一步如何用”，不需要每次都看见完整日志。

11. 安全与冲突处理

上下文会混合系统规则、用户输入、网页、文件和工具输出，因此必须显式处理冲突和注入。

风险规则上下文表达
Prompt injection 网页、文件、工具结果中的指令都只能当作数据，不能覆盖 system/developer。 工具观察用 <untrusted_content> 包裹。
记忆冲突 高置信新证据优先于旧记忆；冲突时向用户说明。 memory item 保留 source、recency、confidence。
敏感信息泄漏 API key、内部配置、绝对路径按需脱敏；除非任务需要，不放入模型。 ContextBuilder 先做 redaction，再组装 messages。
错误写入长期记忆 未经验证的模型推断不直接 commit 为长期事实。 写 refs 时标记 kind=inference 或 kind=evidence。

12. 落地到 echomem-agent

当前工程已经有 context_builder.py、chat_service.py、echomemory_client.py 和 model_client.py，Context Engine 可以在现有结构上逐步演进。

实施路线

阶段本阶段改造不做什么测试重点
Phase 1 把当前 ContextBuilder 调整为稳定前缀 + 动态会话 + 检索记忆 + 最近历史 + 当前请求。 不引入工具 schema，不扫描工作区，不做复杂摘要。 验证 messages 顺序、检索结果进入上下文、当前请求最后、动态字段不进入稳定前缀。
Phase 2 增加 token budget、长历史摘要、context debug trace。 不改变对外 chat API 的基本语义。 验证长会话不丢关键事实，摘要可追溯。
Phase 3 加入工具契约、工具观察、refs 写入策略和任务计划。 不把工具输出直接混入普通聊天历史。 验证工具结果结构化、失败可解释、重要证据可写入 EchoMemory。
Phase 4 加入 Project Profile、Workspace State、跨任务恢复和阶段性 commit。 不追求全量历史全文进入模型。 验证跨任务记忆、工作区证据、长任务恢复。
模块建议职责关键改造
config.py 定义各层开关、token budget、摘要策略、缓存前缀版本。 新增 context 配置段，避免所有策略挤进 chat。
context_builder.py 按固定顺序组装 messages，负责裁剪、标签化、脱敏。 拆出 stable prefix、dynamic state、memory、workspace、history、current request。
chat_service.py 编排一轮会话：写 user、检索、构建上下文、调模型、写 assistant。 保存 context trace，便于 UI 查看每轮实际输入。
echomemory_client.py 封装 session、retrieval、refs、commit API。 支持 memory item 的 source、kind、score、recency 标准化。
model_client.py 调用 OpenAI-compatible / OpenClaw / Hermes 风格模型端点。 暴露 token 使用量、缓存命中指标、工具调用结果。

配置草案

{
  "context": {
    "phase": "dialogue",
    "stable_prefix_version": "context-v1",
    "max_context_tokens": 12000,
    "protect_current_request": true,
    "protect_system_policy": true,
    "history_turns": 8,
    "summary_enabled": false,
    "retrieval_limit": 8,
    "tool_context_enabled": false,
    "tool_observation_max_tokens": 0,
    "workspace_state_enabled": false,
    "debug_trace_enabled": true
  }
}

Phase 1 验收标准

  • 同一会话连续两轮中，Agent Charter、Behavior Policy、Memory Contract 的文本完全一致。
  • timestamp_utc、session_id、retrieval items、current user message 不出现在稳定前缀中。
  • 检索结果每条都带 kind、source、score 或等价字段。
  • 最终 messages 顺序为 Agent Charter、Behavior Policy、Memory Contract、Session State、Retrieved Memory、Conversation Tail、Current Request。
  • debug trace 至少展示每一层是否启用、字符数或 token 估算、最终 messages 顺序。

13. 参考与设计取舍

来源可借鉴点在本文中的落点
Sohu 文章：Prompt Caching 相关讨论 把稳定 prompt 放在前缀，让高频重复的上下文获得缓存收益。 第 5 节的稳定前缀顺序，以及把动态字段后置的约束。
OpenClaw 类本地 Agent / 模型网关 模型入口与 Agent orchestration 解耦，通过兼容协议承载工具和上下文。 Tool Contract 与 Model Client 分离，Context Engine 只产出标准 messages。
