EchoMemory Agent Design

Agent Web Playground

本文定义一个真实 Agent 对话场景驱动的本地 Web 页面。页面模拟用户与 Agent 的对话、工具调用和提交归档， 通过 EchoMemory 标准对外接口写入数据，并通过 runtime / inspection 调测接口实时查看 EchoMemory 内部状态是否正确变化。

设计目录：docs/design/agent 代码目录：src/agent 默认端口：8765 主场景：Agent Chat 首版验证：D01-D03
目标 原则 对话场景 动作映射 页面结构 标准接口 调测接口 代码布局 实施计划 验收

1. 目标

目标不是做一个“内部对象操作台”，而是做一个真实 Agent 使用 EchoMemory 的最小可视化 playground。 用户在页面中看到的是对话、工具调用和提交按钮；EchoMemory 侧看到的是标准 service/API 调用和可检查的 runtime 状态。

真实场景

以对话为中心

首屏就是 Agent 对话，而不是模块列表。打开新对话会触发 session open，发送消息会触发 add message。

标准接口

Agent 调用公开能力

Agent 页面不直接写文件系统，不调用内部 archive helper，只通过 EchoMemory 对外接口完成 session、message、ref、commit。

可观察

调测接口看内部状态

页面右侧通过 runtime、filesystem tree、event trace 等 inspection API 查看系统是否真的写入、归档和投递。

2. 设计原则

原则要求原因
Agent first 页面功能从 Agent 的自然动作出发：新对话、发消息、工具调用、提交当前对话。 验证体验更接近真实集成方式，避免变成低层 API 调试器。
标准接口优先 写入类操作只能通过 EchoMemory 对外接口，不能通过调测接口绕过业务服务。 确保页面验证的是未来 SDK/Agent 真正会依赖的合同。
观察与操作分离 Agent 动作负责写入；inspection API 只负责查看 runtime、tree、manifest、event trace。 保留信息隐藏与权限边界，同时让内部执行结果足够直观。
不提前伪造未来能力 D04/D05 未完成前，只显示 not_ready 状态，不模拟记忆提取和召回。 避免页面看似完整但代码能力不存在。

3. Agent 对话场景

首版页面围绕一个“Agent 帮用户完成任务”的最小闭环：打开新对话，用户发出需求，Agent 回复，Agent 可模拟一次工具调用， 最后用户点击 commit 验证 session 归档。 

1. 用户点击「新对话」
   -> Agent App 调用 EchoMemory session.open(user_id, agent_id, session_id)

2. 用户发送第一条消息
   -> Agent App 调用 session.add_message(role="user", content=...)

3. Agent 生成回复
   -> Agent App 调用 session.add_message(role="assistant", content=...)

4. Agent 模拟工具调用
   -> Agent App 调用 session.add_ref(kind="tool_call", ...)
   -> Agent App 可追加 tool role message 或 assistant message 描述结果

5. 用户点击「Commit」
   -> Agent App 调用 session.commit()
   -> EchoMemory 写入 archive 并发布 session.committed 事件

6. 页面 inspection 区自动刷新
   -> 查看 runtime 状态、echo://sessions tree、archive.json、event trace
这里的“首次提交”不是一个隐藏批处理。新对话必须显式调用 session.open，首条用户消息必须显式调用 session.add_message。这样页面能清楚展示每一步对 EchoMemory 状态的影响。

4. 页面动作与 EchoMemory 调用映射

页面动作EchoMemory 标准调用页面反馈调测观察
新对话 SessionService.open(user_id, agent_id, session_id) 生成当前对话上下文，显示 session scope。 echo://sessions/{session_id}/current/session.json 出现。
发送用户消息 SessionHandle.add_message(role="user", content) 对话流新增用户气泡。 current/messages.jsonl 新增一行。
Agent 回复 SessionHandle.add_message(role="assistant", content) 对话流新增 Agent 气泡。 current/messages.jsonl 新增 assistant 记录。
模拟工具调用 SessionHandle.add_ref(kind="tool_call", name, content, mime_type) 对话流显示工具卡片。 current/refs/ 出现 ref 文件，refs.jsonl 新增记录。
Commit SessionHandle.commit() 显示 SessionCommitResult。 history/archive_001 出现，event trace 出现 session.committed。

5. Web 页面结构

页面应像一个小型 Agent 客户端，而不是 API 表单集合。左侧是对话主工作区，右侧是 EchoMemory 状态观察区。

Agent Chat

对话主区

包含 user_id、agent_id、session_id、新对话按钮、消息输入框、Agent 回复输入框、模拟工具调用按钮和 Commit 按钮。

EchoMemory Inspector

状态观察区

实时展示 runtime snapshot、session tree、archive manifest、event trace、最后一次 API response。

区域控件/内容说明
对话配置栏 user_id、agent_id、session_id、新对话。 新对话只调用 session.open，不会自动 commit。
消息流 用户气泡、Agent 气泡、工具卡片。 每条展示对应的 EchoMemory 调用状态，失败时保留错误。
输入区 发送用户消息、添加 Agent 回复、模拟工具调用。 按钮触发标准接口调用，调用成功后刷新 inspector。
提交区 Commit 按钮、commit result。 用于验证归档边界；返回值不展示内部 archive URI。
观察区 Runtime、目标 URI 文件系统视图、Archive、Events、Raw JSON。 支持树状展开和平铺展开；点击文件通过只读接口查看内容，不提供写文件能力。

6. Agent 使用的标准对外接口

Agent 页面中的业务动作应调用 EchoMemory 标准对外接口。若当前代码只完成 service 层接口，Web 实现可以先在 agent server 内部调用 SessionService，但路径和语义应按未来标准 API 设计，避免调测接口成为事实标准。

方法路径语义底层当前实现
POST /api/sessions/open 打开新对话或恢复已有对话。 SessionService.open(...)
POST /api/sessions/{session_id}/messages 追加 user / assistant / tool message。 SessionHandle.add_message(...)
POST /api/sessions/{session_id}/refs 追加工具调用、工具结果、搜索结果或 artifact。 SessionHandle.add_ref(...)
POST /api/sessions/{session_id}/commit 提交当前对话为 archive。 SessionHandle.commit()

示例流程

POST /api/sessions/open
{
  "user_id": "alice",
  "agent_id": "demo-agent",
  "session_id": "chat-001"
}

POST /api/sessions/chat-001/messages
{
  "role": "user",
  "content": "帮我整理一下 D03 的提交方案"
}

POST /api/sessions/chat-001/refs
{
  "kind": "tool_call",
  "name": "read_design_doc",
  "mime_type": "application/json",
  "content": "{\"path\":\"docs/design/features/03.session-commit.html\"}"
}

POST /api/sessions/chat-001/commit
{}

7. EchoMemory 调测接口

调测接口只用于观察内部执行是否成功，不承担 Agent 写入动作。路径建议使用 /agent/inspect/**， 既表明它服务于 agent playground，又避免和正式 /api/** 混淆。

方法路径用途约束
GET /agent/inspect/runtime 查看 runtime mode、feature readiness、workspace zones。 复用 runtime_payload，不暴露 workspace 绝对路径。
GET /agent/inspect/fs/tree?uri=echo://sessions/{session_id} 只读查看 session 当前目录和 history 目录。 只允许 echo://，限制 max_depth。
GET /agent/inspect/fs/read?uri=... 读取 session.json、messages.jsonl、archive.json。 只读，不提供 write/delete/move。
GET /agent/inspect/events 查看本地 recording event sink 收到的事件。 agent playground only，不作为正式 event API。
调测接口可以显示 archive manifest 和 event payload 中的内部 URI，因为它的定位是本地观察；但这些字段不能反向进入 SessionCommitResult 或 Agent 标准写入接口。

src/
├── echomem/
│   ├── server.py                  # 正式 EchoMemory HTTP 服务
│   ├── runtime/
│   └── service/

    ├── __init__.py
    ├── agent_server.py             # Agent playground server，默认 127.0.0.1:8765

目录用途边界

tests/test_agent_*.py 存放 Agent Playground 的 HTTP 黑盒测试。 使用临时 workspace 和端口 0。

9. 实施计划

步骤修改内容原因
1 创建 docs/design/agent 和 src/agent。 按 agent playground 定位组织设计与代码，且保证后续可以迁移到独立项目。
2 实现 agent server 首页，首屏为对话页面。 让验证从真实 Agent 对话开始，而不是从内部模块开始。
3 实现 /api/sessions/open 和 /api/sessions/{id}/messages。 完成“新对话 + 首条消息”的首次提交前置动作。
4 实现 /api/sessions/{id}/refs。 模拟工具调用和工具结果，验证 multiple refs 能进入 session。
5 实现 /api/sessions/{id}/commit 与 Commit 按钮。 手动触发归档，观察 archive 和事件投递。
6 实现 /agent/inspect/runtime、FS tree/read、events。 实时查看 EchoMemory 内部执行是否成功。
