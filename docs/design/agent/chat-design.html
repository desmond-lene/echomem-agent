EchoMem Agent / Chat Runtime Design

Agent Chat 深度改造设计

本文定义 echomem-agent 的会话能力升级方案：开放工程配置、接入真实大模型、通过标准 HTTP 方式连接 EchoMemory， 并按标准 Agent 流程完成检索增强、上下文拼接、模型调用、session 写入与归档。

目标工程：echomem-agent 设计文件：docs/design/agent/chat-design.html 模型入口：OpenAI-compatible / OpenClaw 记忆入口：EchoMemory HTTP API
目标 开放配置 大模型接入 EchoMemory 对接 上下文拼接 会话流程 接口设计 文件规划 验收

1. 改造目标

当前 echomem-agent 更像 EchoMemory HTTP API 的 playground：页面负责发起 session open、add message、add ref、commit， assistant 回复仍主要依赖手工输入。下一阶段要把它升级成真正的 Agent Chat Runtime。

配置开放

运行参数不写死

EchoMemory 地址、模型 provider、base URL、model、temperature、上下文预算、检索开关和 session 策略全部由配置决定。

真实模型

接入本机 OpenClaw

通过 OpenAI-compatible chat completions 协议对接本机 OpenClaw 中已配置好的大模型，避免在 agent 内绑定特定 SDK。

标准 Agent

检索增强上下文

每轮用户输入先写入 EchoMemory session，再调用 Retrieval API，按 system、memory、history、user 的顺序拼接上下文并调用模型。

设计约束：echomem-agent 仍保持独立工程边界，不 import echomem.* 内部模块，只通过 EchoMemory HTTP API 通信。

2. 开放配置设计

工程需要支持配置文件、环境变量和启动参数三层输入。优先级从高到低为：命令行参数、环境变量、配置文件、默认值。 配置应放在 agent 工程内，不依赖 EchoMemory workspace 的内部配置。

3. 大模型接入方式

模型层抽象成 ChatModelClient，默认实现为 OpenAICompatibleChatClient。 这样本机 OpenClaw、OpenAI-compatible 网关、后续真实 OpenAI API 或其它兼容服务都可以走同一个协议。 

class ChatModelClient(Protocol):
    def complete(self, request: ChatCompletionRequest) -> ChatCompletionResult:
        ...

@dataclass(frozen=True)
class ChatCompletionRequest:
    messages: tuple[ChatMessage, ...]
    model: str
    temperature: float
    max_tokens: int

@dataclass(frozen=True)
class ChatCompletionResult:
    content: str
    raw: Mapping[str, Any]

HTTP 调用契约

POST {model.base_url}/chat/completions
Authorization: Bearer {model.api_key}
Content-Type: application/json

{
  "model": "openclaw-default",
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."}
  ],
  "temperature": 0.3,
  "max_tokens": 1024
}

第一阶段只要求 non-streaming 响应，后续再扩展 SSE streaming。模型调用失败时，Agent 应返回结构化错误， 不应该写入 assistant message；用户消息是否保留由 session 策略决定。

4. EchoMemory 标准对接

echomem-agent 只调用 EchoMemory 对外 HTTP API，不 import runtime、service、filesystem 等内部类。核心接口分为两组： session 写入接口和 retrieval 召回接口。

动作EchoMemory APIAgent 使用方式
打开会话 POST /api/sessions/open 新建 chat 或首次收到消息时调用，获得稳定 scope。
写入用户消息 POST /api/sessions/{session_id}/messages 每轮用户输入先写入 session，role 为 user。
检索上下文 POST /api/retrieval/search 使用当前 query、user_id、agent_id、session_id 获取上下文候选。
写入助手回复 POST /api/sessions/{session_id}/messages 模型成功返回后写入 assistant message。
附加工具结果 POST /api/sessions/{session_id}/refs 工具调用、检索证据、外部文件摘要等作为 ref 写入。
归档提交 POST /api/sessions/{session_id}/commit 手动、轮次阈值或会话结束时 commit，触发 memory engine 消费。

Retrieval 请求示例

POST /api/retrieval/search
{
  "query": "用户这次想让我怎么整理文档？",
  "user_id": "alice",
  "agent_id": "demo-agent",
  "session_id": "chat-001",
  "limit": 6,
  "include_explain": true
}

5. 标准 Agent 上下文拼接

上下文拼接由 ContextBuilder 负责。它不直接调用模型，也不直接写 session，只接收当前会话状态、检索结果和用户输入， 输出符合 chat completions 协议的 messages。

1. System固定角色、能力边界、输出规范。
2. Developer工程级策略、工具说明、EchoMemory 使用约束。
3. Memory来自 RetrievalService 的 ContextItem，按来源和 kind 分组。
4. History最近 N 轮 user/assistant 消息，受 token budget 约束。
5. Tool Context当前轮工具结果或 refs 摘要。
6. User当前用户原始输入，永远放在最后。

拼接模板

[
  {"role": "system", "content": chat.system_prompt},
  {"role": "developer", "content": "你可以使用 EchoMemory 检索结果，但不得编造未给出的证据。"},
  {
    "role": "system",
    "content": "<memory>\n- [preference/simple] 用户喜欢简洁技术文档\n</memory>"
  },
  {"role": "user", "content": "上一轮用户消息"},
  {"role": "assistant", "content": "上一轮助手回复"},
  {"role": "user", "content": "当前用户输入"}
]

上下文预算规则

  • 优先保留 system/developer/current user，不允许被截断。
  • memory item 按 score、kind、recency 和 evidence 完整度排序，超过预算时从低优先级裁剪。
  • history 只保留最近 N 轮；如果仍超预算，先摘要早期 history，再裁剪。
  • retrieval explain 仅进入调试输出，不默认进入模型上下文。

6. 会话主流程

User submit message
  -> AgentChatService.ensure_session(user_id, agent_id, session_id)
  -> EchoMemory POST /api/sessions/{id}/messages role=user
  -> EchoMemory POST /api/retrieval/search
  -> ContextBuilder.build(system + memory + history + current user)
  -> ChatModelClient.complete(messages)
  -> EchoMemory POST /api/sessions/{id}/messages role=assistant
  -> 按策略决定是否 commit
  -> 返回 assistant content、retrieval items、session status

这个流程把 EchoMemory 当作 Agent 的长期记忆和会话事实源，把 OpenClaw/LLM 当作推理生成器。Agent 自身负责 orchestration， 不把记忆写入、检索路由、memory engine 细节耦合进模型调用层。

7. Agent 工程对外接口

Agent server 面向前端提供更高层的 chat API，内部再调用 EchoMemory 和模型。这样页面不用手工发 session/retrieval/model 多个请求。

接口说明是否代理 EchoMemory
GET /Agent Chat 页面。否
GET /agent/config返回脱敏后的当前配置。否
POST /agent/chat标准 Agent 一轮对话：写 user、检索、拼上下文、调模型、写 assistant。间接调用
POST /agent/session/commit显式提交当前 session。是
/api/**保留向 EchoMemory 的低层代理能力，便于调试。是
/agent/inspect/**只读观察 EchoMemory 状态。是

    ├── test_config.py

    ├── test_chat_service.py

模块职责边界
config.py读取 JSON、环境变量和 CLI override。不创建模型和 HTTP client。
echomemory_client.py封装 session、retrieval、commit HTTP API。不 import echomem 内部模块。
model_client.py调用 OpenAI-compatible chat completions。不拼接业务上下文。
context_builder.py按标准 Agent 消息顺序构造 messages。不发 HTTP，不调用模型。
chat_service.py串联 session 写入、retrieval、context、LLM、assistant 写入。不直接读写 EchoMemory 文件系统。

9. 验收标准

用例验证点
test_agent_loads_open_config 配置文件、环境变量和 CLI override 能合并；敏感字段在 /agent/config 中脱敏。
test_model_client_calls_openai_compatible_endpoint 模型 client 使用标准 /v1/chat/completions 请求格式，可指向本机 OpenClaw。
test_chat_uses_echomemory_session_api 一轮 chat 必须调用 session open/add user/add assistant，不绕过 EchoMemory。
test_chat_uses_retrieval_before_model 模型调用前必须调用 /api/retrieval/search，并把返回 items 放入上下文。
test_context_builder_orders_messages 上下文顺序固定为 system、developer、memory、history、tool、current user。
test_chat_does_not_write_assistant_on_model_failure 模型失败时返回错误，不写入伪造 assistant 消息。
完成标准：启动 echomem-agent 后，用户在页面发送消息，Agent 能自动写入 EchoMemory session、召回历史/偏好、 拼接标准 chat messages、调用本机 OpenClaw 模型、写回 assistant 消息，并可手动或按策略 commit session。
