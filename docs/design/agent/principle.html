EchoMemory Engineering Principles

工程设计原则

本文定义 EchoMemory 的代码分层、依赖方向、接口组织、测试替换和设计约束。后续所有 MVP 交付项都应以本文为工程准绳。

1. 总原则

EchoMemory 不是脚本集合，而是一个可长期演进的记忆内核。工程结构必须让业务规则、应用编排和基础设施实现相互隔离， 让本地 MVP、未来插件引擎、生产化存储和单元测试 Fake 都能在同一套边界下自然接入。

领域优先

业务规则不依赖技术细节

记忆、会话、召回、引擎事件等核心概念必须先在领域模型中稳定下来，再由应用服务和基础设施实现。

接口隔离

依赖抽象而不是实现

服务层只依赖接口；LocalFS、SQLite、模型网关、向量检索都必须可以替换为 Fake 或其他实现。

可测试

单元测试不依赖外部环境

默认测试必须可以在无网络、无模型服务、无外部数据库的环境中运行，符合 AIR 原则。

生命周期最小化

数据只进入必要生命周期

能用当前 session 历史解决的问题，不提升为长期记忆；能作为可重建投影存在的内容，不写成事实源。

作用域最小化

默认使用最窄作用域

所有读取、写入、召回和投影都必须优先限定在 user、agent、session、entity 等最小必要 scope 内。

核心判断标准：如果一个模块无法用 Fake 替换外部依赖后独立测试，说明它的边界还不够清晰。

2. 依赖方向规则

依赖方向必须清晰：允许上层调用下层、平层调用；禁止下层调用上层。这里的“调用”包括 import、运行时回调、全局单例访问和隐式依赖。 

API / CLI / HTTP
        ↓
Application Service / Recall / Runtime
        ↓
Domain / Engine Contract / Interfaces
        ↓
Infrastructure Implementations
规则 1：Domain 不知道外部世界。 Domain 层不能 import infra、service、server、cli，也不能读取环境变量、文件、数据库或模型服务。
规则 2：Application 只依赖接口。 Application 层可以编排 MemoryRepository、FileSystem、EmbeddingProvider 等接口，但不能 new 一个 SQLiteMemoryRepository。
规则 3：Infrastructure 实现接口，但不调用上层。 SQLite、LocalFS、FTS5、ModelGateway 实现可以依赖接口定义和领域对象，但不能调用 SessionService、SearchService 或具体 MemoryEngine。
规则 4：跨层通信通过数据和事件，不通过反向调用。 引擎处理完成后返回 EngineResult 或发布标准事件，由上层编排下一步；引擎不直接调用服务层。

3. 接口组织原则

每个 package 下必须定义 interfaces 子 package。接口定义放在所属 package 的 interfaces 中， 接口使用到的共享实体按可见范围决定放置位置。

规则说明示例
每个 package 都有 interfaces 接口是包对外的稳定契约，不能散落在实现文件中。 infra/interfaces/filesystem.py、engine/interfaces/engine.py
接口专用实体可定义为内部类型 如果实体只被该接口内部使用，可以放在同一个接口文件中，降低跳转成本。 HealthCheckResult 只被 HealthCheckable 使用。
共享实体放入 interfaces/entities 如果实体被多个接口、服务或测试使用，应放入接口包下的 entities，避免实现层重复定义。 infra/interfaces/entities/fs_entry.py、recall/interfaces/entities/search_trace.py
契约随交付项就近定义 不要提前冻结一批全局模型。只有被当前交付项真实使用、且跨模块共享的对象，才进入对应 interfaces/entities。 SessionArchive 随 SessionService 定义；EngineEvent 随 EngineRegistry 定义。
引擎内部模型归引擎所有 主仓库不定义 Tree/Graph/Timeline 的内部记忆 schema，只定义引擎运行时必须遵守的边界对象。 MemoryRecord 如仅服务 TreeMemoryEngine，应放在引擎工程内。
接口命名表达能力，不表达实现 接口名应描述职责，而不是具体技术。 用 VectorIndex，不用 FaissClient。
接口粒度遵循 ISP 不要让调用方依赖自己不用的方法。读写、查询、重建、健康检查可以拆成小接口；其中 WritableFileSystem 只表示 infra adapter 的内部写契约，不应直接暴露为通用 FSService API。 ReadableFileSystem、内部 WritableFileSystem、RebuildableIndex
信息隐藏优先 底层对上层只暴露上层必须知道的最少信息。接口返回值不泄露底层模型、文件布局、事件投递状态、幂等键、索引表结构或具体 adapter 细节。 SessionCommitResult 返回 session_id、archive_id、committed_at；不返回 archive_uri、event_id、idempotency_key。
DTO 与领域对象命名：DTO 对象只用于接口输入输出边界；输入统一以 Req 结尾，输出统一以 Rsp 结尾。 领域对象直接按业务用途命名，不增加 Req、Rsp 后缀。DTO 与领域对象必须分组存放： 可以在同一文件中先集中定义 DTO、再集中定义领域对象；也可以拆成 dto 与 entities 子 package。 禁止把传输对象和领域对象混排，避免接口契约、业务语义和实现细节互相污染。

4. 基础设施层原则

基础设施层必须对外暴露接口，理论上支持多种实现。任何外部依赖都必须可以被测试替身替换，以保证单元测试可以在任意环境直接运行。

Runtime 负责依赖注入和实现选择。业务服务不得根据环境变量直接选择实现；它们只接收接口实例。Fake、Stub、Spy、Mock 等测试替身属于测试代码，默认不放入 src 生产包。

5. SOLID 与补充设计原则

原则EchoMemory 中的要求反例
SRP 单一职责框架服务只编排 session、搜索、文件访问和引擎事件投递；记忆提取、生成、更新、遗忘和生命周期管理内化到各 MemoryEngine。框架层写一个巨大记忆生命周期服务，同时做抽取、合并、索引、HTTP 响应和引擎投影。
代码规模受控一个类通常不应超过 300 行，最多不超过 500 行；单个方法通常不超过 40 行。超过限制时应优先按职责拆分为小类、小模块或纯函数。一个 MemoryEngine 类同时包含 prompt 渲染、模型调用、规则抽取、记录写入和 HTTP 适配，导致单文件和单方法持续膨胀。
三方代理原则所有第三方系统、模型服务、外部 API 和 SDK 对接必须封装在 infra 中，并以 XxxProxy 命名。上层只能依赖接口，不能暴露 API key、HTTP header、SDK client、endpoint 拼接、重试和协议细节。TreeMemoryEngine 直接 import urllib.request 调大模型；召回层直接 new 第三方 SDK client；infra 中出现不以 Proxy 结尾的三方对接类。
OCP 开闭原则新增向量库、搜索引擎、记忆引擎时新增 adapter 或 engine，不修改核心服务流程。在 SearchService 中写一堆 if 判断 Qdrant、Milvus、SQLite。
LSP 里氏替换FakeVectorIndex、SQLiteNumpyVectorIndex、QdrantVectorIndex 对调用方表现一致。Fake 返回结构和真实实现不同，导致测试通过但生产失败。
ISP 接口隔离读、写、重建、健康检查拆成合适接口；调用方只依赖需要的能力。所有 infra adapter 都被迫实现一个巨大的 Storage 接口。
信息隐藏上层接口只返回上层决策需要的信息；底层路径、内部事件、幂等键、索引状态等细节留在 manifest、trace 或诊断接口中。commit() 把 echo://sessions/.../messages.jsonl、event_status、SQLite 表名直接返回给 SDK 调用方。
生命周期最小化数据进入的生命周期必须与用途匹配：当前对话上下文保留在 session history，已完成任务进入 archive，长期稳定事实才进入 MemoryEngine，摘要、索引和缓存必须作为可重建派生产物存在。第二轮对话需要上一轮“你好”时先写入长期偏好；或把 L0/L1、BM25、Vector 结果当作事实源保存。
作用域最小化读写和召回默认使用最窄 scope。能限定到 session 的不扩散到 user，能限定到 agent 的不扩散到全局；跨 user、跨 agent、跨 project 的访问必须显式建模和配置。SearchService 在没有明确请求时跨所有用户搜索；TreeMemoryEngine 把一次 session 里的临时偏好直接写成全局用户画像。
最小参数暴露构造函数和方法只接收自己实际需要的最小参数；不要传入大配置、大路径对象或上层运行时对象后只读取其中一个字段。LocalFS(WorkspacePaths) 只为了读取 echo_root；应改为 LocalFS(echo_root)。
稳定依赖前置注入对象的稳定运行时依赖应在构造、加载或 Runtime 装配阶段注入；单次方法调用只传本次输入，避免调用方被迫了解底层上下文。MemoryEngine.process(event, context) 每次都传 fs、projection_root、config，导致召回层也必须知道 engine 运行时细节。
DIP 依赖倒置Application 依赖接口，Runtime 负责注入具体实现。服务层直接 import SQLiteFtsIndex。
DRY 适度复用重复第三次再抽象；抽象必须减少真实复杂度。为了消除两行重复创建难懂的泛型框架。
KISS 简单优先MVP 默认本地、同步优先，异步队列和分布式 adapter 后置。第一个版本就引入复杂消息队列和多租户平台。
YAGNI 不预支实现预留接口和边界，但不提前实现 Resource、Skill、Relation、Console。为尚未进入 MVP 的功能写一整套空实现。
业界实践优先代码设计遵循 Python 社区和后端工程的成熟实践：清晰命名、类型标注、依赖注入、明确错误、可测试边界和最小公开 API。为了短期方便访问私有属性、绕过构造流程、把测试替身放进生产包。
必要注释注释解释设计意图、边界条件和不明显的约束；简单赋值、普通分支和能由命名表达的逻辑不写噪音注释。整段复杂权限判断没有说明；或给 name = uri.name 写“设置名称”这类注释。
简洁实现始终选择能表达意图的最简写法；在合理抽象前提下减少分支、嵌套和重复，优先使用早返回、小函数和数据结构表达规则。为两个分支引入复杂框架；或在 service 中堆叠多层 if/else 处理 adapter 差异。
显式优于隐式所有重要行为通过配置、manifest、trace 和 diff 显式表达。依赖全局变量、魔法目录和隐式环境判断。
No Use-Case Customization (Mandatory)Do not add any code path tailored to a specific example, person name, company name, or a single evaluation sample. Improvements must be implemented as general rules, configurable strategies, or independent reusable modules, and validated by generic tests.Adding branches like if name == "PersonX", if org in {"CompanyY"}, or query-specific shortcut matching for one benchmark question.

6. 测试与 AIR 原则

单元测试必须满足 AIR：Automatic、Independent、Repeatable。测试不应依赖网络、真实模型、外部数据库、机器时间和随机 ID。

AIR 要求工程约束实现方式
Automatic 自动化测试一条命令即可运行，不需要人工准备服务。默认使用 tests/fake 中的测试替身和临时目录；集成测试用 marker 单独开启。
Independent 独立性测试之间不能依赖顺序、共享状态或真实用户目录。每个测试创建独立 runtime、FakeClock、SequenceIdGenerator。
Repeatable 可重复相同输入必须得到相同输出。FakeModelGateway 返回固定摘要和 embedding；时间和 ID 固定。
 
from tests.fake.runtime import make_fake_runtime


def test_memory_search_with_fake_runtime():
    runtime = make_fake_runtime()
    session = runtime.session_service.open(user_id="alice", agent_id="codex")

    session.add_message("user", "我喜欢简洁的中文技术文档")
    session.commit()

    result = runtime.search_service.search(
        "用户喜欢什么文档风格？",
        user_id="alice",
        agent_id="codex",
    )

    assert result.items[0].evidence_uri.startswith("echo://users/alice")
真实 SQLite、真实模型、真实 HTTP 服务测试属于集成测试；它们可以存在，但不能成为基础单元测试通过的前置条件。

测试替身归属

EchoMemory 只允许两类 Fake：一类是 infra adapter 的 Fake，用来替代文件系统、索引、模型、时钟、ID 等外部依赖； 另一类是 MemoryEngine 的 Fake，用来保证主框架在没有真实三方引擎时仍可验证事件和召回闭环。 Application 层规则组件和业务编排组件不定义 Fake，例如不为 IntentAnalyzer、RecallOrchestrator、SearchService 创建 Fake。

原则要求示例
Fake 范围受限Fake 只用于 infra adapter 和 MemoryEngine。业务规则组件应直接测试真实实现，必要时通过数据样例覆盖分支。允许 FakeFS、FakeModelGateway、FakeMemoryEngine；不允许 FakeIntentAnalyzer。
测试替身不进生产包Fake 默认放在 tests 内，避免 src 混入仅为测试存在的实现。tests/fake/filesystem.py 提供 FakeFS。
按测试用途组织共享 Fake 放在 tests/fake 子包；仅单个测试文件使用的简单 Fake 可以就近定义在测试文件中。tests/fake/model.py、tests/fake/memory_engine.py。
Fake 遵守接口契约Fake 必须实现同一接口、权限和错误语义；不能为了让测试容易而改变调用方看到的行为。FakeFS 和 LocalFS 对非法 URI、越权写入抛出同类错误。
Runtime 只装配允许的 Fake生产 runtime 默认装配真实实现；测试装配工厂只能从允许的两类 Fake 注入替身，业务代码不得直接 import 测试替身。tests/fake/runtime.py 可注入 FakeFS 和 FakeMemoryEngine。

7. 代码评审检查清单

检查项必须满足
分层是否只存在上层调用下层或平层调用？是否存在 infra 调 service、domain 调 infra 的反向依赖？
接口新增外部依赖是否先定义接口？接口是否放在对应 package 的 interfaces 下？
实体接口共享实体是否放入 interfaces/entities？是否避免了跨实现重复定义？
DTO接口输入 DTO 是否统一使用 Req 后缀、输出 DTO 是否统一使用 Rsp 后缀？DTO 与领域对象是否已分组存放，避免混排？
测试是否在 tests/fake 提供必要测试替身？单元测试是否能在无网络、无外部数据库、无真实模型的环境运行？
SOLID是否出现巨大 service、巨大 interface、硬编码实现选择、违反替换原则的 Fake？
代码规模类是否通常控制在 300 行内、最多不超过 500 行？方法是否通常控制在 40 行内？超限代码是否已按职责拆分？
三方代理第三方模型、外部 API、SDK 和网络协议是否全部封装在 infra 的 XxxProxy 中？上层是否只依赖代理接口？
可观测关键行为是否能通过 trace、diff、metrics 或日志解释？
可重建索引、L0/L1、缓存是否都能从事实源重建？是否没有把派生产物当事实源？
生命周期当前信息是否进入了最短可满足需求的生命周期？是否避免把临时上下文误提升为长期记忆？
作用域读写、召回和投影是否使用最窄 scope？是否避免无意跨 user、agent、session 或 entity 污染？
范围是否把非 MVP 能力提前做成复杂实现？是否能用接口预留而不是提前平台化？
EchoMemory engineering principles. Keep dependencies honest, tests boring, and boundaries useful.
